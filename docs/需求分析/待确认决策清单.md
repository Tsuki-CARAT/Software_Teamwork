## 2. 必须优先确认

这些决策会影响所有服务、前端调用和后续 OpenAPI/数据库设计，应先确认。

### D1. 认证方式

问题：API 认证采用哪种方式？

建议选项：

| 选项 | 说明 | 影响 |
| --- | --- | --- |
| A. Cookie Session | 浏览器体验简单，适合前后端同站部署；SSE 和下载 URL 处理较顺 | 需要 CSRF 防护和 SameSite/CORS 策略 |
| B. Bearer Token/JWT | 适合前后端分离、移动端或第三方调用 | 前端需要安全保存 token，SSE 传递 header 需实现细节 |
| C. 二者兼容 | 浏览器用 Cookie，开放 API 用 Bearer Token | 实现成本更高，但扩展性最好 |

建议默认：首期选 A 或 C。若只面向 Web 应用并走同一网关，A 更简单。

需要同步更新：

- 三份 API 契约的“认证与权限”部分。
- SSE 连接和下载 URL 的鉴权说明。
- `auth` 服务契约。

### D2. API 网关前缀

问题：外部 API 是否统一使用 `/api/v1/...`？

建议选项：

| 选项 | 说明 | 影响 |
| --- | --- | --- |
| A. `/api/v1/...` | 当前契约暂按此写法 | 版本清晰，后续 v2 可并行 |
| B. `/api/...` | 更短，但版本管理弱 | 版本演进时需要额外策略 |
| C. 服务名直出 | 如 `/knowledge/...`、`/qa/...` | 不利于网关统一和前端封装 |

建议默认：A。

### D3. 异步任务机制

问题：文档处理、向量化、报告生成使用哪种任务机制？

建议选项：

| 选项 | 说明 | 影响 |
| --- | --- | --- |
| A. PostgreSQL 任务表 + worker 轮询 | 简单可靠，业务状态和任务状态统一持久化 | 高并发能力有限，但首期足够 |
| B. Redis 队列 + PostgreSQL 持久化状态 | 响应更快，适合轻量队列 | 要处理 Redis 丢失和幂等 |
| C. 专用消息队列 | 更可靠、吞吐更高 | 增加部署复杂度 |

建议默认：首期选 A 或 B。无复杂吞吐要求时，A 最稳；需要更流畅任务调度时，B。

### D4. 会话历史持久化

问题：智能问答会话历史保存在哪里？

原始需求提到“本地持久化”，但系统需要多轮对话、刷新恢复和审计能力。

建议选项：

| 选项 | 说明 | 影响 |
| --- | --- | --- |
| A. 服务端 PostgreSQL 持久化 | 支持多端、审计、统计、历史恢复 | 需要后端会话/消息 API |
| B. 前端本地存储 | 实现快，但无法多端同步和服务端统计 | 与权限、审计、引用追溯不匹配 |
| C. 双写 | 服务端为主，前端缓存改善体验 | 复杂度略高 |

建议默认：A，前端可做缓存但不作为业务真相。

### D5. 报告支撑材料归属

问题：报告支撑材料作为哪类资源建模？

建议选项：

| 选项 | 说明 | 影响 |
| --- | --- | --- |
| A. 知识库文档的一种 `support_material` 类型 | 复用知识库处理链路最彻底 | 可能和“报告专用材料”业务概念混在一起 |
| B. 独立资源，但复用 file/knowledge 处理和检索能力 | 业务语义清晰，报告模块只引用 | 需要额外 materials API |
| C. 完全归属报告生成模块 | 报告实现简单 | 容易重复文档处理和检索逻辑，不利于解耦 |

建议默认：B。当前契约按 B 表达，同时保留与知识检索的复用关系。

## 3. 高优先级确认

### D6. OCR 与文档解析后端

问题：首期使用什么文档解析/OCR 后端？

建议选项：

| 选项 | 说明 | 影响 |
| --- | --- | --- |
| A. 外部 API | 快速落地，效果取决于供应商 | 需要密钥、限流、失败重试 |
| B. 本地解析 + 外部 OCR | 普通文档本地处理，图片/扫描件外部 OCR | 平衡成本和效果 |
| C. 全本地化 | 适合内网部署 | 部署和模型维护成本高 |

建议默认：B。如果首期时间紧，先 A。

### D7. 模型供应商与配置

问题：LLM、Embedding、Rerank 默认供应商和配置方式是什么？

当前需求中提到 SiliconFlow 云端 API 参考地址，但未明确必须使用。

建议选项：

| 选项 | 说明 |
| --- | --- |
| A. 首期默认 SiliconFlow，配置可替换 |
| B. 多供应商抽象，首期至少实现一个供应商 |
| C. 完全本地模型优先 |

建议默认：B，落地时先实现 SiliconFlow provider。

### D8. 数据分析意图

问题：智能问答中的“数据分析”意图是否本期实现？

建议选项：

| 选项 | 说明 | 影响 |
| --- | --- | --- |
| A. 本期不实现，只保留枚举和拒绝/提示 | 降低范围，避免多出一个模块 |
| B. 本期实现基础 Excel/表格分析 | 需要单独数据分析 API 和沙箱能力 |
| C. 并入报告生成的数据处理 | 可能造成模块边界不清 |

建议默认：A。

### D9. 数据权限粒度

问题：是否需要按组织、电厂、专业控制知识库、文档、报告访问？

建议选项：

| 选项 | 说明 |
| --- | --- |
| A. 首期只做角色级 RBAC |
| B. 增加知识库/报告所有者和共享范围 |
| C. 加组织、电厂、专业多维权限 |

建议默认：B。C 可作为后续增强，除非项目验收明确要求。

### D10. 审计日志范围

问题：哪些操作必须记录审计日志？

建议首期至少覆盖：

- 登录失败/成功。
- 原文下载。
- 报告下载。
- 删除知识库、文档、模板、报告。
- 修改模型配置和系统配置。
- 权限和角色变更。

待确认：审计日志是否需要管理后台查询页面，还是先仅后端记录。

## 4. 模块内细节确认

### 4.1 知识管理

| 编号 | 问题 | 建议 |
| --- | --- | --- |
| K1 | 知识库删除是软删除还是硬删除？ | 首期软删除，后台异步清理 Qdrant/MinIO |
| K2 | embedding 维度变化后如何处理 Qdrant collection？ | 新建版本化 collection，旧数据后台迁移 |
| K3 | 文档处理失败后保留几次错误历史？ | 至少保留最近一次，建议保留多次处理记录 |
| K4 | 文档标签是自由键值对还是预定义字典？ | 首期自由键值对，后续可加字典 |

### 4.2 智能问答

| 编号 | 问题 | 建议 |
| --- | --- | --- |
| QA1 | 流式中断是否保存部分回答？ | 保存部分内容并标记 `cancelled` 或 `failed` |
| QA2 | 无权限知识库如何处理？ | 默认过滤不可见知识库，同时在调试信息中给出 filtered 数量 |
| QA3 | 引用是否保存快照？ | 保存引用片段快照，避免源文档删除后历史回答不可追溯 |
| QA4 | 思考过程展示是否展示原始推理链？ | 不展示原始 chain-of-thought，只展示处理过程摘要 |

### 4.3 报告生成

| 编号 | 问题 | 建议 |
| --- | --- | --- |
| R1 | 模板文件格式是否限定 DOCX？ | 首期限定 DOCX，后续支持 JSON/Markdown 模板配置 |
| R2 | 固定大纲来自哪里？ | 首期数据库模板配置或内置模板，长期放到模板配置 |
| R3 | 重新生成单章是否覆盖用户编辑？ | 必须显式选择是否覆盖，默认不覆盖 |
| R4 | 图/表编号模式是否首期实现？ | 作为加分项，接口保留字段 |
| R5 | DOCX 统一样式是否首期必须？ | 若报告完成度为考核重点，建议 P1 提前 |

## 5. 建议确认顺序

1. D1 认证方式。
2. D2 API 前缀。
3. D3 异步任务机制。
4. D4 会话历史持久化。
5. D5 报告支撑材料归属。
6. D6-D10 高优先级技术和权限边界。
7. K/QA/R 模块细节。

完成前五项后，三份 API 契约即可进入实现前定稿；后续细节可在具体开发任务中继续收敛。
